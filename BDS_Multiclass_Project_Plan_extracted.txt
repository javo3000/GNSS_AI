AI-Based BDS Positioning Reliability Assessment
for Disaster Response Navigation
Full Project Plan
Multiclass reliability classification with Random Forest, XGBoost and LightGBM. BDS-only scope, dataset already extracted, no code.
1. Project Overview
This plan replaces the earlier M2DGR-based version. M2DGR is no longer part of the project. The scope is now limited to a single data source: BDS observations from IGS/MGEX ground stations, processed through RTKLIB. The feature table has already been extracted from this data, so this plan starts from the point where labelling begins and carries through to the final report.
The objective is unchanged in spirit. The project still asks whether a BDS position estimate can be trusted, and it still connects that answer to a navigation decision for an autonomous rescue robot. The robot is not simulated and not built. The two IGS stations act as a stand-in for a receiver under open-sky and obstructed conditions, and the model output represents the kind of signal a rescue robot would use to decide how much to rely on BDS at a given moment.
Three tree-based models are trained and compared on the same data and the same split: Random Forest, XGBoost, and LightGBM. The comparison is the main technical contribution of this version of the project, since the original brief only asked for two models.
2. Research Aim and Objectives
Aim: build a three-class BDS positioning reliability classifier from RTKLIB-derived ground station features, and identify which of three tree-based models gives the safest detection of unreliable positioning.
Generate reliability class labels from the true horizontal positioning error already present in the dataset.
Train Random Forest, XGBoost, and LightGBM on identical features, identical class labels, and identical train-test split.
Compare the three models on overall accuracy, per-class recall, and Class 2 recall in particular.
Rank the features driving the classification decision, and check whether the three models agree on which features matter most.
Translate the best model's output into a rescue robot decision table.

Every model in this project is trained and evaluated on this same set of ten features, so any difference in results comes from the model itself and not from a difference in inputs.
5. Label Generation Methodology
Each epoch receives a reliability class from 0 to 2, based on the true horizontal error and, for Class 2, on supporting satellite geometry conditions.

1

true_h_error_m between 3 and 10 m
BDS is degraded. Robot slows and cross-checks.
2

true_h_error_m > 10 m, or num_satellites < 4, or pdop > 10

Apply these rules to every row in all four datasets and print the class distribution for each dataset separately before merging. JFNG should be heavily Class 0. URUM should carry more Class 1 and some Class 2. If Class 2 is empty across all four datasets after applying the 10 metre threshold, lower it to 5 metres and reapply. This is a methodological choice, not an error, and the chosen threshold must be stated clearly in the report.
6. Train-Test Split Strategy
The training set stacks three datasets: JFNG 18 December, JFNG 10 December, and URUM 10 December. These cover open-sky conditions and obstructed conditions on a normal day, so the model learns what reliable, degraded, and poor BDS performance looks like across environments.
The test set is URUM 18 December only, the earthquake date in obstructed terrain. It is never touched during training. This gives the project a result that is tied to a real event rather than an arbitrary random split, and it is the hardest available test of all three models.
Add a station_date identifier column to each file before merging, so that rows can be traced back to their source after the four datasets are combined.
7. Model Training Plan
All three models are trained on the same ten input features, the same reliability_class target, the same training rows, and the same random seed where applicable. This keeps the comparison fair: any difference in results reflects the model, not the data.

7.3 LightGBM (Third Tree-Based Model)
LightGBM is a second gradient boosting implementation, built around a leaf-wise growth strategy rather than the level-wise growth used by most other boosting libraries. It is included as a third model because it sometimes behaves differently from XGBoost on minority class recall, even though both are boosting methods. Comparing three models rather than two checks whether strong Class 2 recall is a general result or specific to one library.

9. Model Comparison Framework
The model selected for the rescue robot decision framework is chosen using the following order of priority, not by accuracy alone.
Highest recall for Class 2, since a missed unsafe reading is the most dangerous failure mode.
If two models tie closely on Class 2 recall, prefer the one with higher overall accuracy.
If accuracy is also close, prefer the simpler or faster model for a future real-time deployment.
Alongside the metric comparison, check whether the three models agree on which features matter most. If num_satellites, pdop, or pos_uncertainty consistently rank at the top across Random Forest, XGBoost, and LightGBM, that is a stronger and more trustworthy finding than a single model's importance ranking on its own.

14. Limitations and Future Work
Both stations are static. The project does not capture how a moving rescue robot would behave, since motion introduces its own position uncertainty that a fixed station does not have.
Only two dates are covered. Seasonal, weather, and ionospheric variation across a longer period is not tested.
The dataset is BDS-only single point positioning. No RTK fixed solution, no IMU, and no LiDAR are part of this scope, since M2DGR has been removed.
Future work: extend the framework to a moving platform with IMU fusion, test additional boosting libraries such as CatBoost, and validate the labelling thresholds against a longer time series before any real deployment.
